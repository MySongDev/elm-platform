# API 版本与兼容性策略

本文档定义后端 API、Swagger 文档和前端生成类型之间的兼容规则。

## 兼容变更

以下变更通常可以直接合并，但仍需更新 Swagger 和测试：

- 新增可选字段。
- 新增不影响旧客户端的接口。
- 新增枚举值，且旧客户端能安全忽略。
- 扩展分页、筛选、排序参数，并保持默认行为不变。

## Breaking Change

以下变更视为破坏性变更：

- 删除字段。
- 重命名字段。
- 修改字段类型或含义。
- 修改枚举值含义、删除枚举值或让旧值失效。
- 修改错误响应结构。
- 修改认证、权限或租户隔离要求。
- 修改分页、排序、筛选参数的默认行为。
- 修改接口路径、HTTP method 或状态码语义。

## 必做动作

涉及 API 契约变更的 PR 必须完成：

- 更新 DTO、Swagger decorators 和接口文档。
- 运行 `pnpm api:generate`。
- 提交 `packages/api-types/openapi.json` 和 `packages/api-types/generated/index.d.ts` 的预期变化。
- 在 PR 中说明前端、移动端或外部调用方的迁移方案。
- 为高价值路径补充后端契约测试或前端 API adapter 测试。

## 版本策略

默认优先保持当前 API 兼容。确实需要 breaking change 时，优先采用以下方式之一：

- 新增并行字段，保留旧字段至少一个发布周期。
- 新增新接口版本，例如 `/api/v2/...`。
- 使用 feature flag 或配置控制新旧行为。
- 在 PR 和 changelog 中写明下线时间和迁移步骤。

## Review 要点

Reviewer 需要确认：

- Swagger、DTO、生成类型和调用方是否一致。
- 旧客户端是否仍能工作。
- 错误响应是否保持可解析。
- 权限、认证、租户隔离变化是否有审计说明。
- API 类型漂移是否是预期结果。
