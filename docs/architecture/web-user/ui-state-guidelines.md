# 页面状态与移动端 UI 规范

这份规范用于保持项目页面体验一致，也方便后续扩展测试和设计评审。

## 页面状态

所有依赖接口的页面至少考虑四种状态：

- `loading`：首次加载使用骨架屏或局部 loading。
- `empty`：接口成功但列表为空，使用明确的空状态文案。
- `error`：接口失败，给出重试入口。
- `success`：展示业务内容。

推荐优先使用已有的 `BaseState`、`ShopListSkeleton`、`van-empty` 和全局 loading store，不在每个页面重复写状态分支。

## 登录拦截

页面级权限使用路由 `meta.requiresAuth`。操作级权限使用 `useAuthRedirect()`，例如搜索按钮、加载更多、进入商家详情、进入支付流程。

## 移动端布局

- 页面主色保持蓝色 `#3190e8` 或项目变量 `$blue`。
- 列表项高度、边框、字号保持稳定，避免动态内容导致跳动。
- 按钮文字不超过一行，长文本使用 `@include text-ellipsis`。
- 高优先级操作放右侧或底部固定区，返回入口放左侧。
- 卡片只用于独立业务块，不嵌套卡片。

## 文案规范

- 错误提示使用可行动文案，例如“获取定位失败，请重新定位”。
- 空状态说明用户下一步可以做什么。
- 高曝光页面不保留乱码、调试文案和无意义注释。

## 测试建议

状态逻辑优先测试纯函数和 composable：字段格式化、本地状态、地址展示、定位归一化、权限跳转。
