# 项目亮点与面试讲解稿

这份文档用于面试前快速梳理项目价值，帮助把“写了什么”转化成“为什么这样设计”。

## 1. 路由与权限

项目使用 `src/router/modules` 按业务拆分路由，主路由通过 `import.meta.glob` 自动收集模块。这样新增业务页面时只需要添加一个路由模块，不需要频繁改主路由文件。

登录权限通过 `meta.requiresAuth` 标记。全局守卫只负责页面级权限，页面内操作负责更细粒度的交互拦截，例如未登录搜索、未登录加载更多、未登录进入商家详情。

面试表达：

> 我把权限分成“页面能不能进”和“操作能不能触发”两层。路由守卫解决页面级访问控制，页面内部则处理业务动作，避免把所有规则堆到一个全局守卫里。

## 2. HTTP 请求封装

HTTP 层统一封装 Axios，负责：

- 请求/响应拦截
- 经纬度参数注入
- 全局 loading
- 业务错误提示
- 网络错误重试
- 错误日志收集

这样页面组件只关注“请求什么数据”和“如何展示”，不用重复处理 loading、错误提示和公共参数。

面试表达：

> 外卖业务很多接口依赖经纬度，因此我把位置注入放到请求拦截器里，让业务 API 保持干净，同时也保证所有接口使用同一套位置来源。

## 3. Pinia 状态设计

项目的共享状态按领域拆分：

- `store-user`：登录态、用户信息、退出登录
- `store-locations`：经纬度、geohash、城市与地址
- `store-address`：收货地址列表
- `store-loading`：全局 loading

这些状态跨页面共享或需要持久化，因此放入 Pinia。页面内短暂 UI 状态仍保留在组件中。

面试表达：

> 我没有把所有状态都放进 store，只有跨页面共享、有持久化需求或具备领域意义的状态才进入 Pinia。

## 4. 页面状态体系

项目中有 `BaseState`、骨架屏、空态、错误态、loading 组件。它们用于统一页面常见状态，减少各页面重复写 `loading/error/empty` 分支。

面试表达：

> 企业项目中接口失败、空数据和加载态是高频场景。如果每个页面各写一套，很容易体验不一致，所以我用基础状态组件统一这些模式。

## 5. 移动端列表体验

商家列表使用滚动容器和触底加载，配合 loading、finished 防止重复请求。页面还支持返回顶部和滚动位置恢复，解决移动端长列表常见体验问题。

面试表达：

> 列表加载不是简单的 `scroll` 事件，而是围绕 loading、finished、去重和滚动容器做了一套可复用逻辑，避免重复拉取和滚动错乱。

## 6. 地址管理

收货地址作为用户中心的独立功能，入口放在 Profile 页主导航中，而不是隐藏在账号信息里。地址列表页采用独立白底头部和移动端列表布局，新增地址继续复用现有表单校验能力。

面试表达：

> 地址是高频业务入口，不应该藏在账户信息页。我把它移动到用户中心主导航，并让地址页独立表达自己的业务结构。

## 7. 支付流程

项目包含支付宝支付相关本地服务和前端支付页。支付流程被拆成下单、跳转支付、支付结果查询，方便本地调试和后续替换支付渠道。

面试表达：

> 支付相关逻辑我没有直接散落在商家页，而是抽成独立 API 和页面流程，降低支付能力对业务页面的侵入。

## 8. 可以继续升级的点

- 核心业务逐步迁移 TypeScript。
- 增加 Vitest 单元测试和 Playwright E2E。
- 增加 CI：lint、test、build。
- 把商家页、地址页、首页定位进一步拆成 composable。
- 建立完整设计 token，统一字体、间距、颜色和状态组件。
