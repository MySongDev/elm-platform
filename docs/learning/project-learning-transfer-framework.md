# 项目学习方法的通用迁移框架

## 1. 核心结论

`AI 时代广度和深度的取舍` 这套方法可以迁移到其他项目中。

但需要注意：

> 通用的不是 Vue、NestJS、Redis 这些具体技术，而是背后的学习和判断方法。

也就是说，它不是只适用于 Elm 项目，而是一套更通用的项目学习框架：

```text
以项目为主线，
以问题为入口，
以架构为目标，
以 AI 为辅助。
```

这套方法可以迁移到前端项目、后端项目、全栈项目、管理后台、移动端、小程序、AI 应用项目等不同类型的项目。

---

## 2. 通用部分是什么

这套方法可以抽象为四层：

```text
1. 建立项目地图。
2. 用真实问题决定深挖方向。
3. 用 L1 / L2 / L3 分配学习精力。
4. 用 RFC 和复盘训练架构判断。
```

项目不同，技术栈不同，但这四层方法可以复用。

---

## 3. 第一层：先建立项目地图

做任何项目，第一步都不是马上深挖某个技术，而是先搞清楚项目整体。

需要回答：

```text
这个项目解决什么业务问题？
核心用户是谁？
核心流程是什么？
有哪些核心模块？
技术栈是什么？
数据从哪里来，到哪里去？
用户主要操作路径是什么？
哪些地方最容易出问题？
```

对于 Elm 项目，项目地图可能是：

```text
用户端
管理后台
NestJS 后端
Prisma
PostgreSQL
Redis
JWT
Swagger
```

换成博客系统，项目地图可能是：

```text
文章发布
评论
标签
用户登录
后台管理
Markdown 渲染
SEO
图片上传
```

换成 AI Chat 项目，项目地图可能是：

```text
会话管理
消息流式输出
模型调用
上下文压缩
文件上传
RAG 检索
用量计费
权限控制
```

形式不同，但第一步一样：

> 先建立地图，不要一上来钻细节。

---

## 4. 第二层：用真实问题决定深挖方向

不要问：

```text
这个项目用了什么技术，我就全部学到底。
```

而要问：

```text
这个项目真正复杂的地方在哪里？
哪些问题如果处理不好，会影响长期维护？
哪些决策会影响后续扩展？
```

### 4.1 电商项目

值得深挖的通常不是“怎么写商品卡片”，而是：

```text
购物车状态如何设计？
订单状态机如何设计？
支付失败如何恢复？
库存扣减如何避免超卖？
优惠券叠加规则如何建模？
```

### 4.2 博客项目

值得深挖的通常不是“怎么写文章列表”，而是：

```text
Markdown 如何安全渲染？
草稿、发布、下架状态如何设计？
SEO 和动态内容如何取舍？
标签和分类如何建模？
图片资源如何管理？
```

### 4.3 AI Chat 项目

值得深挖的通常不是“怎么调接口”，而是：

```text
上下文如何裁剪？
流式响应如何中断和重试？
历史消息如何存储？
RAG 检索结果如何排序？
模型调用失败如何降级？
token 和费用如何控制？
```

### 4.4 管理后台项目

值得深挖的通常不是“怎么画表格”，而是：

```text
权限模型如何设计？
动态路由如何生成？
表格查询条件如何配置化？
批量操作如何设计回滚？
审计日志如何记录？
```

项目不同，但方法一样：

> 找出项目里会影响维护、扩展、稳定性、成本的问题，这些才值得深挖。

---

## 5. 第三层：L1 / L2 / L3 的通用迁移

L1 / L2 / L3 可以用于任何项目。

### 5.1 L1：怎么写

L1 是功能实现层。

例如：

```text
怎么写一个按钮？
怎么写一个接口？
怎么写一个表单？
怎么写一个 SQL 查询？
怎么写一个 API 调用？
```

这些问题 AI 很擅长。

在任何项目里，L1 都可以大量借助 AI。

学习目标是：

```text
看懂。
会改。
能验证。
不盲信。
```

而不是把所有 API 都背下来。

### 5.2 L2：为什么这样写

L2 是原理和模式层。

例如：

```text
为什么要分层？
为什么要用状态管理？
为什么要做缓存？
为什么要加索引？
为什么要写 DTO？
为什么要有鉴权中间件？
```

L2 是识别代码质量的基础。

需要知道：

```text
这个模式解决了什么问题？
不这样做会有什么后果？
它的边界在哪里？
```

### 5.3 L3：该不该这样写

L3 是架构决策层。

例如：

```text
这个项目现在该不该引入缓存？
这个功能该前端做还是后端做？
这个模块现在该不该抽象？
这个项目该不该拆服务？
这个页面该 SSR 还是 CSR？
这个接口该 REST 还是 GraphQL？
这个权限该做到菜单级、按钮级还是接口级？
```

这类问题最有价值，也最难被 AI 完全替代。

因为不同项目的上下文完全不同：

```text
团队人数不同。
业务阶段不同。
维护周期不同。
性能要求不同。
交付时间不同。
用户规模不同。
历史代码不同。
```

AI 可以给建议，但最终判断要靠项目负责人自己。

## 6. 新项目学习模板

以后接触一个新项目，可以直接套这个模板。

### 6.1 第一步：项目地图

问题：

```text
这个项目解决什么业务？
核心用户是谁？
核心流程是什么？
有哪些模块？
技术栈是什么？
前端、后端、数据库、缓存、部署分别是什么？
```

输出：

```text
项目模块图
核心业务流程图
技术栈清单
主要数据流
```

### 6.2 第二步：广度索引

问题：

```text
项目里出现了哪些我不熟悉的技术？
它们分别解决什么问题？
哪些只是工具？
哪些影响架构？
哪些当前必须懂？
哪些可以先知道概念？
```

输出：

```text
必须深挖
适度理解
只需了解
暂时跳过
```

### 6.3 第三步：复杂点识别

问题：

```text
这个项目最复杂的业务是什么？
最容易出 bug 的地方是什么？
最影响扩展的模块是什么？
最影响性能的链路是什么？
最影响协作的边界是什么？
```

输出：

```text
项目深挖主题清单
```

### 6.4 第四步：L1 / L2 / L3 拆分

对每个主题拆分：

```text
L1：怎么实现？
L2：为什么这样实现？
L3：当前项目该不该这样设计？
```

例如缓存：

```text
L1：Redis 怎么 set/get？
L2：缓存穿透、击穿、雪崩是什么？
L3：当前项目哪些接口值得缓存？缓存不一致能不能接受？
```

### 6.5 第五步：写小 RFC

每遇到一个关键决策，写一页：

```text
背景
问题
约束
方案 A
方案 B
方案 C
推荐方案
成本
风险
回滚策略
后续观察指标
```

这一步最训练 L3。

### 6.6 第六步：项目复盘

项目完成或阶段结束后，复盘：

```text
当初判断对了吗？
哪里低估了复杂度？
哪里过度设计了？
哪里抽象太早了？
哪里应该提前做治理？
哪些 AI 生成内容后来证明有问题？
```

复盘会形成长期复利。

## 8. 最重要的迁移原则

以后做任何项目，都不要把“项目用了什么技术”当成学习路线。

应该把：

```text
项目真正复杂在哪里？
```

当成学习路线。

例如项目用了：

```text
Vue
React
NestJS
Redis
Docker
PostgreSQL
Kafka
OpenAI API
```

这不代表每个都要深挖。

真正要问的是：

```text
这个项目的核心复杂度在哪里？
```

如果核心复杂度是权限，那就深挖权限。

如果核心复杂度是支付，那就深挖支付状态机。

如果核心复杂度是性能，那就深挖性能链路。

如果核心复杂度是 AI 成本，那就深挖 token 管理和上下文策略。

如果核心复杂度是多人协作，那就深挖模块边界和工程规范。

---

## 9. 迁移公式

迁移到其他项目时，只需要替换三样东西：

```text
1. 替换项目上下文。
2. 替换核心复杂问题。
3. 替换 L3 训练题。
```

Elm 项目里的深挖题是：

```text
HTTP 治理
权限系统
动态路由
缓存策略
组件分层
Monorepo 共享包
```

换成 AI Chat 项目，可能变成：

```text
会话状态
流式输出
上下文压缩
RAG 检索
模型降级
token 成本控制
```

换成博客系统，可能变成：

```text
内容模型
编辑器
权限
SEO
评论
资源管理
```

底层方法不变：

```text
广度建地图；
深度解问题；
L1 借助 AI；
L2 用来审查；
L3 形成判断；
项目复盘积累经验。
```

---

## 10. 最终总结

这套方法可以作为以后做项目的通用学习框架。

但要记住：

> 通用的是方法，不是清单。

不要把某个项目中的 Vue、NestJS、Redis 当成固定路线。

真正固定的是这套思考方式：

```text
先看项目地图；
再识别复杂度；
用广度知道有哪些路；
用深度判断走哪条路；
用 L1 / L2 / L3 分配学习精力；
用 RFC 和复盘训练架构能力；
用 AI 提高产出，但不外包判断。
```

如果能把这套方式迁移到不同项目中，学习项目的速度和深度都会明显提升。
