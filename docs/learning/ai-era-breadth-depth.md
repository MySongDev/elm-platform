# AI 时代广度和深度的取舍：从 T 型到梳子型

## 1. 核心结论

AI 时代，学习的重点不再是把所有技术都“学会”，而是要重新区分广度和深度的价值：

- **广度**：建立技术地图，知道有哪些方案、解决什么问题、适合什么场景、边界在哪里。
- **深度**：形成决策能力，知道当前项目该选哪种方案、为什么选、代价是什么、失败后如何回滚。

一句话总结：

> 广度决定你不迷路，深度决定你能到达别人到不了的地方。

对当前 Elm 全栈项目来说，学习策略应该是：

```text
以项目为主线，
以问题为入口，
以架构为目标，
以 AI 为辅助。
```

不要把目标设为“再学多少技术”，而应该转向：

```text
我的项目还有哪些真实企业级问题没有被系统性解决？
```

这个问题会直接决定哪些内容应该深挖，哪些内容只需要保持广度了解。

---

## 2. 为什么从 T 型转向梳子型

传统的 T 型能力模型强调：

```text
一个主专业很深，
其他领域略懂一些。
```

但在 AI 辅助编程越来越强的背景下，很多“略懂”和“查资料式能力”已经被 AI 大幅降低门槛。单纯知道很多技术名词，已经不能形成明显竞争力。

更适合当前阶段的是“梳子型”能力：

```text
多个技术齿：知道它们解决什么问题、适合什么场景、有什么成本；
几个核心齿：在真实项目中深入落地，并能解释架构取舍。
```

对于当前 Elm 项目，可以理解为：

```text
前端主齿：Vue / TypeScript / 组件 / 状态 / 路由 / HTTP / 工程化
后端协作齿：NestJS / Prisma / PostgreSQL / Redis / Auth / Swagger
架构治理齿：模块边界 / 权限模型 / 错误体系 / 性能 / 可维护性
AI 协作齿：会提问 / 会审查 / 会验证 / 会让 AI 产出工程级结果
```

梳子型不是要求每个齿都成为专家，而是每个齿都至少能回答：

```text
1. 它解决什么问题？
2. 什么时候适合用？
3. 不用它会怎样？
4. 引入它会带来什么成本？
5. 在当前项目里值不值得用？
```

## 3. 广度：从“学会”转向“索引化”

过去的广度学习容易变成：

```text
我要学会 React。
我要学会 NestJS。
我要学会 Redis。
我要学会 Docker。
我要学会微前端。
```

AI 时代更有效的广度学习应该是索引化：

```text
我知道某个技术存在；
我知道它解决哪类问题；
我知道它适合什么规模；
我知道它会带来什么成本；
我知道项目真的需要时可以借助 AI 和文档快速深入。
```

例如 Redis 的有效广度不是只记住：

```text
Redis 是一个高性能 key-value 数据库。
```

而是知道：

```text
解决什么：缓存、热点数据、验证码、限流、分布式锁、队列辅助。
不用会怎样：高频接口可能重复查数据库，性能和稳定性受影响。
适合什么：有高频访问、共享状态、短期数据、异步任务辅助需求时。
成本是什么：部署、缓存一致性、过期策略、穿透/击穿/雪崩治理。
```

有效广度的标准是：

> 能用三句话讲清楚“这是什么、解决什么问题、什么时候用”。

---

## 4. 深度：从“细节”转向“决策层”

AI 可以生成很多代码片段，也可以解释很多概念。但它不能完全替代你做项目决策，因为它缺少你的真实上下文：业务阶段、团队能力、历史包袱、维护成本、未来目标。

深度不等于死磕所有源码，也不等于背 API。深度更接近：

```text
面对一个真实问题，
能拆清楚约束，
能比较多个方案，
能判断收益和成本，
能设计落地路径，
能预判失败边界，
能解释为什么不选另一种方案。
```

可以分为三个层次：

| 层级 | 内容 | AI 替代程度 | 学习策略 |
| --- | --- | --- | --- |
| L1 语法/工具 | API、配置、简单 CRUD、页面组件 | 高 | 不重记忆，用时查，借助 AI |
| L2 原理/模式 | 框架机制、工程模式、常见设计思想 | 中 | 理解原理，能识别 AI 输出质量 |
| L3 架构/权衡 | 选型、边界、风险、维护性、演进路径 | 低 | 重点训练，形成自己的判断 |

核心原则：

> L1 交给 AI，L2 用来理解和审查，L3 必须自己训练。

## 5. L1 / L2 / L3 在 Elm 项目中的具象化

### 5.1 L1：功能实现层

在 Elm 项目里，L1 类问题包括：

```text
写一个登录页面。
写一个商品列表组件。
写一个 NestJS Controller。
写一个 Prisma 查询。
写一个 Pinia store。
写一个表单校验。
写一个 axios 请求。
```

这些问题通常需求明确，答案相对固定，AI 生成准确率较高。你需要能看懂和修改，但不需要把主要学习时间消耗在记忆细节上。

### 5.2 L2：原理和模式层

L2 类问题包括：

```text
Vue 响应式为什么能追踪依赖？
Pinia 为什么适合按业务模块拆分？
Vue Router 动态路由为什么要 reset？
Axios 拦截器为什么适合做统一错误处理？
NestJS 为什么要分 Controller / Service / Module？
Prisma 和直接 SQL 的取舍是什么？
JWT 鉴权流程是什么？
Redis 缓存穿透、击穿、雪崩分别是什么？
```

L2 的目标不是炫技，而是让你具备审查能力：

```text
AI 生成的代码是否合理？
这个封装是否过度？
这个方案是否违背框架习惯？
这个错误处理是否会污染业务组件？
```

### 5.3 L3：架构决策层

L3 类问题更加贴近真实工程：

```text
用户端和管理后台是否应该共享一套 API 类型？
是否应该引入 OpenAPI 自动生成前端请求代码？
后台动态路由应该以后端菜单为准，还是前端维护白名单？
HTTP 错误处理应该统一到 shared/api，还是每个业务模块自己处理？
用户端是否需要 SWR 缓存？哪些接口应该缓存，哪些不能缓存？
Redis 应该用在哪里？商品详情缓存？验证码？登录状态？限流？
权限系统只做到菜单级，还是要做到按钮级、接口级？
Prisma schema 如何设计，后期扩展商家、订单、评价是否容易？
Monorepo 里的 packages 现在是否需要抽 shared 包，还是暂时不抽？
前端是否应该全面 TypeScript 化？如果是，先改用户端还是管理端？
```

这些问题没有唯一标准答案，必须结合项目目标、维护成本和演进方向来判断。

当前项目已经包含用户端、管理后台、后端服务、数据库、缓存、认证、Swagger、Monorepo 等元素，因此不缺广度入口。

更合适的学习比例是：

```text
60%：围绕当前 Elm 项目深挖
25%：补全前后端协作广度
15%：了解行业趋势和新技术
```

应该深挖：

```text

权限系统
动态路由
状态管理
错误处理
接口类型化
组件边界
业务模块划分
后端模块设计
数据库模型设计

```

目标不是成为纯后端，而是成为懂后端协作的前端/全栈开发者。

需要理解：


### 6.3 15%：了解行业趋势和新技术

## 7. 当前项目的深挖方向

### 7.1 HTTP 请求治理

要关注的不只是“怎么请求接口”，而是：

```text
请求实例如何组织？
token 如何注入？
401 / 403 / 业务错误 / 网络错误如何区分？
错误提示在哪里触发？
请求重试、去重、缓存是否默认开启？
哪些逻辑放 HTTP 层，哪些逻辑放 service 层？
如何避免组件里散落 try/catch 和错误码判断？
```

### 7.2 权限系统

不仅要实现登录，还要理解权限模型：

```text
登录态如何保存？
JWT 过期后如何处理？
后台菜单权限、路由权限、按钮权限如何分层？
后端接口权限和前端权限如何对应？
动态路由失败时如何兜底？
```

### 7.3 动态路由和菜单协议

管理后台已经有动态路由建设基础，适合深挖：

```text
后端菜单数据如何映射成前端路由？
component key 为什么必须受控？
路由 reset 的边界是什么？
无权限、菜单缺失、组件不存在时如何降级？
```

### 7.4 组件和业务分层

用户端和管理端都适合训练组件边界：

```text
哪些是纯 UI 组件？
哪些是业务组件？
哪些逻辑应该抽 composable？
哪些逻辑应该留在页面中？
什么时候抽象，什么时候不抽象？
```

### 7.5 后端模块和数据模型

后端适合训练“业务模型到数据模型”的能力：

```text
Controller / Service / DTO 的职责边界是什么？
Prisma schema 如何支持后续业务扩展？
哪些查询应该直接查数据库，哪些应该封装领域服务？
哪些数据需要缓存，哪些数据不应该缓存？
```

## 8. 只需保持广度了解的内容

下面这些内容可以先作为技术地图，不需要马上深挖：

```text
微前端：知道适合多团队、多子应用、独立部署的复杂场景。
SSR / SSG：知道适合首屏、SEO、内容型站点或特定性能目标。
WebAssembly：知道适合计算密集型场景，不适合普通页面开发。
低代码：知道适合表单、配置化后台、重复业务流程，不适合所有复杂业务。
边缘计算：知道适合低延迟、靠近用户的静态/轻计算场景。
跨端框架：知道适合多端复用，但会引入平台差异和调试成本。
```

判断原则：

> 只要它不能解决当前项目的真实问题，就先保持广度了解。

## 9. L3 架构训练题

可以定期围绕当前项目写一页小 RFC，训练架构判断能力。

### 9.1 HTTP 治理 RFC

问题：

```text
用户端和管理端是否应该共用 HTTP 封装思想？
错误处理应该在哪一层？
401、403、业务错误、网络错误如何区分？
请求重试、缓存、去重是否应该默认开启？
哪些接口不能缓存？
```

输出：

```text
HTTP 治理方案
错误分类表
请求策略说明
失败场景
后续优化路线
```

### 9.2 权限系统 RFC

问题：

```text
权限应该分为菜单级、路由级、按钮级还是接口级？
前端权限和后端权限如何对应？
动态路由和后端菜单协议如何保证稳定？
权限变更后如何刷新路由和状态？
```

输出：

```text
权限模型图
前后端职责划分
路由注册流程
风险点和回滚策略
```

### 9.3 缓存策略 RFC

问题：

```text
哪些接口适合 Redis 缓存？
哪些接口适合前端 SWR？
哪些数据必须实时？
缓存失效如何处理？
缓存错误是否应该影响主流程？
```

输出：

```text
缓存分层方案
接口缓存清单
失效策略
一致性风险说明
```

### 9.4 组件设计 RFC

问题：

```text
哪些组件是 UI 组件？
哪些组件是业务组件？
哪些逻辑应该抽 composable？
什么时候不应该过早抽象？
组件 props 如何设计才稳定？
```

输出：

```text
组件分层规范
示例改造
反例说明
抽象边界原则
```

### 9.5 Monorepo 共享包 RFC

问题：

```text
现在 packages 是否应该抽 shared？
哪些东西值得共享？
API 类型是否共享？
工具函数是否共享？
过早共享会不会增加耦合？
```

输出：

```text
共享包引入时机判断
候选共享内容
不共享的理由
未来演进路径
```

## 10. 每周学习安排

如果每周有 10 小时学习时间，可以参考：

```text
6 小时：围绕 Elm 项目深挖真实问题
2.5 小时：补前后端协作相关广度
1.5 小时：阅读行业趋势、技术雷达、AI 工具变化
```

### 10.1 项目深挖的学习形式

```text
改造一个真实模块。
写一页 RFC。
对比两个方案。
复盘一次历史改动。
给某个模块补测试。
整理一次“为什么这样设计”。
```

### 10.2 广度学习的学习形式

```text
了解概念。
看一个最小 demo。
总结适用场景。
列出成本和边界。
判断当前项目是否需要。
```

### 10.3 趋势学习的学习形式

```text
只建立索引，不马上深挖。
关注它解决的问题，而不是追新名词。
除非能落到当前项目，否则不投入大量时间。
```

## 11. 判断是否值得深挖的公式

遇到一个新技术或新方向时，先问：

```text
1. 它能解决我当前项目里的真实问题吗？
2. 如果不用它，会有什么明显损失吗？
3. 它引入后会增加多少复杂度？
4. 这个复杂度现在值得承担吗？
5. 我能不能解释为什么用它，而不是因为 AI 推荐？
```

如果答案是：

```text
能解决当前真实问题；
不用会明显变差；
复杂度可控；
收益大于成本；
我能讲清楚原因。
```

那就值得深挖。

否则，保持广度了解即可。

更简洁的判断标准：

```text
当前问题：深挖。
未来方向：了解。
面试高频：适度深入。
项目核心：必须做透。
炫技名词：先放一边。
```

## 12. 总结

AI 时代，真正贬值的是：

```text
记 API。
背概念。
写样板代码。
查文档。
生成简单组件。
写普通 CRUD。
```

真正升值的是：

```text
定义问题。
拆解复杂业务。
判断方案成本。
设计代码边界。
发现维护风险。
做架构取舍。
验证 AI 输出。
把混乱需求变成清晰模型。
```

所以当前阶段最适合的路线是：

```text
把 Elm 项目从“功能型项目”升级为“工程型项目”。
```

最终目标不是证明自己“学过很多技术”，而是证明自己能够：

```text
遇到一个混乱需求，
能拆清楚；
写出能维护的代码；
知道什么时候该简单；
知道什么时候该复杂；
还能解释清楚为什么。
```

这就是 AI 时代更值得追求的深度。
